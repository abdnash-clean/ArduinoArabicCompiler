# Arduino Arabic Compiler - Project Analysis

## 1. Overview
This repository implements an Arabic-inspired compiler front-end for Arduino-style programs. It is built in Python and includes:
- An ANTLR-based lexer/parser for an Arabic DSL
- AST construction via a visitor pattern
- Semantic analysis with scope, type checking, and built-in Arduino symbols
- A partial LLVM IR generator using `llvmlite`
- Optional AST visualization support using Graphviz

The compiler accepts Arabic source code and produces LLVM IR for Arduino targets.

## 2. Repository Structure

- `main.py` - compiler driver that normalizes Arabic text, parses input, builds AST, runs semantic analysis, and prints IR.
- `requirements.txt` - Python dependencies: `antlr4-python3-runtime`, `llvmlite`.
- `test_arduino.txt` - sample Arabic Arduino program used by `main.py`.
- `frontend/` - lexer/parser grammar and AST builder.
- `semantic/` - semantic analyzer, symbol table, type system, and built-in function/constant registry.
- `backend/` - LLVM IR generator.
- `ast_dir/` - AST node definitions and visitor interface.
- `ast_visualizer.py` - optional AST visualization implementation.
- `tools/` - utility scripts for ANTLR alias setup.

## 3. Frontend

### Grammar
- `frontend/ArArduinoParser.g4` defines the language syntax.
- Supports:
  - `import` statements
  - variable declarations
  - function declarations
  - special Arduino entrypoints: `اعداد` (setup) and `تكرار` (loop)
  - conditionals (`لو`), loops (`طالما`), `ارجع`, `اقطع`, and `استمر`
  - arithmetic, comparison, logical, and bitwise expressions
  - function calls and assignments

### AST Builder
- `frontend/ast_builder.py` builds an AST from parse tree nodes.
- Uses visitor classes generated by ANTLR.
- Constructs node types from `ast_dir/nodes.py`.
- Includes preprocessing of assignment operators like `+=`, `++`, and `--` by desugaring to `AssignNode` and `BinOpNode`.

### Normalization
- `main.py` includes `normalize_arabic_text()` to:
  - normalize Unicode NFC
  - remove Arabic diacritics and tatweel (ـ)
  - normalize hamza variants (`أ`, `إ`, `آ` → `ا`)

## 4. AST Layer

### Node classes
- `ast_dir/nodes.py` defines AST node dataclasses for:
  - `ProgramNode`
  - `VarDeclNode`, `FuncDeclNode`, `BlockNode`
  - `AssignNode`, `FuncCallNode`, `IfNode`, `WhileNode`, `ReturnNode`
  - `BinOpNode`, `UnaryOpNode`
  - literals: `NumberNode`, `StringNode`, `CharNode`, `BoolNode`
  - `IdNode`, `ImportNode`, `BreakNode`, `ContinueNode`

### Visitor interface
- `ast_dir/visitor_interface.py` declares abstract `visit_*` methods for each node.
- Both semantic analyzer and IR generator implement this interface.

## 5. Semantic Analysis

### Type System
- `semantic/types_system.py` defines:
  - `IntType`, `FloatType`, `StringType`, `BoolType`, `VoidType`, `ErrorType`
  - singletons: `INT_TYPE`, `FLOAT_TYPE`, `STRING_TYPE`, `BOOL_TYPE`, `VOID_TYPE`, `ERROR_TYPE`

### Symbols and Environment
- `semantic/symbols.py` defines `Symbol`, `VariableSymbol`, `FunctionSymbol`, and `ConstantSymbol`.
- `semantic/environment.py` provides nested scopes with `define()` and `resolve()`.

### Built-ins
- `semantic/builtins_registry.py` registers core Arduino-like functions and constants:
  - Constants: `عالي`, `منخفض`, `مدخل`, `مخرج`
  - Functions: `وضع_الطرف`, `اكتب_رقمي`, `اقرأ_رقمي`, `اكتب_تناظري`, `اقرأ_تناظري`, `انتظر`, `الزمن_الحالي`
- Also defines a library table for imports like `سيريال`.

### Semantic Visitor
- `semantic/semantic_analyzer.py` performs:
  - symbol definition and duplicate checks
  - variable declaration type checks
  - expression type inference and arithmetic promotion
  - function declaration and return type validation
  - `لو` condition boolean checking, `طالما` loop boolean checking
  - `اقطع`/`استمر` validation within loops
  - error collection with source line and column metadata

### Behavior
- Allows implicit conversion from `صحيح` to `عشري`.
- Uses `ERROR_TYPE` poisoning to prevent cascaded errors after one failure.
- Tracks function return type inside nested environment.

## 6. Backend / LLVM IR Generation

### IR Generator
- `backend/ir_generator.py` is intended to generate LLVM IR using `llvmlite`.
- Creates an LLVM module targeted at `avr-atmel-none`.
- Declares built-in external functions for Arduino wrappers:
  - `c_digitalWrite`, `c_pinMode`, `c_delay`
- Supports global and local variable allocation.
- Translates:
  - binary arithmetic and comparison operations
  - `if` statements with conditional branches
  - `while` loops with loop blocks
  - variable loads, stores, and constants

### Current coverage and gaps
- `visit_FuncCallNode`, `visit_BoolNode`, `visit_ReturnNode`, `visit_CharNode`, `visit_ImportNode`, `visit_StringNode`, and `visit_UnaryOpNode` are currently empty or incomplete.
- `visit_WhileNode` contains partial loop generation but may not correctly position the initial entry block or follow proper LLVM structure in all cases.
- The IR generator relies on AST node types with semantic resolved types, but some cases are not fully handled.

## 7. AST Visualization
- `ast_visualizer.py` can render the AST to PNG via Graphviz.
- Implements full visitor methods for AST nodes and adds node/edge relationships.
- Not currently active in `main.py`, but the code is present and can be enabled.

## 8. Main Driver
- `main.py` orchestrates compilation steps:
  1. Read `test_arduino.txt`.
  2. Normalize Arabic text.
  3. Lex and parse using ANTLR-generated `ArArduinoLexer` and `ArArduinoParser`.
  4. Build AST with `ASTBuilderVisitor`.
  5. Run semantic analysis with `SemanticAnalyzerVisitor`.
  6. Report semantic errors or generate LLVM IR with `LLVMIRGenerator`.
- If semantic errors exist, the compiler reports them and stops.

## 9. Sample Program
- `test_arduino.txt` demonstrates:
  - global LED pin variable
  - `اعداد`/`تكرار` Arduino entrypoints
  - pin mode setup
  - loop with counter, condition, delay, and digital writes
  - use of built-in constants `عالي` and `منخفض`

## 10. Dependencies
- `antlr4-python3-runtime==4.13.2`
- `llvmlite==0.47.0`

## 11. Known limitations and risks

### Incomplete features
- The backend IR generator is not fully implemented for all AST node types.
- Function calls currently lack a complete codegen path.
- String and char literal code generation is missing.
- `return` code generation is unimplemented.
- Imported library symbol handling is semantic-only and does not generate corresponding IR.

### Language coverage
- No support yet for arrays, structs, or advanced Arduino libraries beyond built-in functions.
- Grammar includes imports, but actual import handling remains incomplete.
- No code generation target beyond LLVM IR string emission.

### Potential runtime issues
- `main.py` assumes `test_arduino.txt` exists in the working directory.
- `ir_generator.py` may crash for unsupported nodes or unresolved variables.

## 12. Recommended next steps
1. Complete missing IR visitor methods in `backend/ir_generator.py`.
2. Add support for function calls and `return` instructions.
3. Expand semantic analysis for imported library symbols and `Serial`-like APIs.
4. Add tests for parser, semantic analysis, and IR generation using sample input files.
5. Consider adding a CLI interface and output file option for generated LLVM IR.
