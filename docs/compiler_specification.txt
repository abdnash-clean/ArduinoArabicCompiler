Optimized tool selection## Project Description for IDE Integration

This project is an Arabic-language Arduino compiler that translates Arabic source into LLVM IR and ultimately AVR firmware. The core components are already present, so IDE integration should focus on exposing those compiler stages cleanly instead of reimplementing them.

### Key integration points

- main.py
  - Current entry point that runs the full pipeline and prints output.
  - For IDE support, this should be refactored into a reusable API layer.

- Compiler stages
  - Lexer / parser: ArArduinoLexer.py, ArArduinoParser.py
  - AST builder: ast_builder.py
  - Semantic analysis: semantic_analyzer.py
  - IR generation: ir_generator.py
  - Optimizer / AVR backend: optimizer.py, avr_target.py, linker.py

- Diagnostics
  - Syntax errors come from ANTLR error listeners on the lexer/parser.
  - Semantic errors are reported by the semantic analyzer.
  - The IDE should merge these into a single diagnostics list with line/column metadata.

### What the IDE should use

- A facade like `compiler_api.py` that exposes:
  - `tokenize(source)` for syntax highlighting
  - `parse(source)` for parse diagnostics
  - `analyze(source)` for semantic diagnostics and AST
  - `generate_ir(source)` for raw LLVM IR
  - `optimize_ir(ir_raw)` for optimized IR
  - `full_compile(source, upto=...)` for build workflows (ASM, object, hex)

- Fast editor feedback
  - Run lexer/parser/semantic passes on edit
  - Return diagnostics for squiggles
  - Keep the source of truth inside the compiler pipeline

- Build/flash integration
  - Use the existing AVR backend and toolchain rather than a separate build system
  - Support explicit build steps on demand, not on every keystroke

### Useful IDE features from existing code

- AST visualization
  - The project already has AST construction and ast_visualizer.py
  - The IDE can show AST panels from the same AST object used by the compiler

- Diagnostics and AI mentor
  - The IDE can leverage the same error listener pattern and semantic error reporting
  - Existing docs mention an AI mentor workflow, which is ideal for a rich IDE experience

- Sample targets
  - blink.txt, serial_hello.txt, and panic_test.txt are useful test cases for integration

### Practical integration focus

- Do not duplicate parser or semantic logic in the IDE
- Wrap compiler stages in functions that return data instead of printing
- Keep frontend and backend artifact generation separate from the UI
- Use a JSON / subprocess bridge if the IDE is not Python-native
- Surface editor diagnostics, IR, optimized IR, AVR assembly, and build logs as panels

### Why this matters

The compiler is the engine; the IDE is the shell. Integration is successful when the IDE:
- reuses the project’s lexer/parser/semantic pipeline,
- shows diagnostics from the same error sources,
- presents IR/ASM generated by the actual backend,
- and invokes build/flash only through the real compiler stages.
